# Синхронное межсервисное взаимодействие в Spring Boot (2026)

## Введение

В этом уроке мы создадим четыре небольших сервиса и на их примере разберём три актуальных технологии синхронного взаимодействия в микросервисной архитектуре. Все сервисы используют **Spring Boot 3.4** и имеют модель, REST-контроллер и сервисный слой.

> **Важное отступление.** Если вы встречали уроки, где для межсервисного взаимодействия используется `RestTemplate` — знайте, что эта технология официально объявлена устаревшей в Spring Framework 7.0 (ноябрь 2025), будет помечена `@Deprecated` в Spring Framework 7.1 (ноябрь 2026), а в 8.0 — удалена полностью. Ей на смену пришёл `RestClient`, который мы и будем использовать. Подробнее — в блоге Spring: [The state of HTTP clients in Spring](https://spring.io/blog/2025/09/30/the-state-of-http-clients-in-spring/).

### Архитектура

```
┌──────────────┐     HTTP      ┌──────────────┐
│  RestClient  │──────────────▶│              │
│  (порт 8081) │               │              │
└──────────────┘               │    Source    │
┌──────────────┐     HTTP      │  (порт 8080) │
│ HTTP Interface│──────────────▶│              │
│  (порт 8082) │               │   H2 in-mem  │
└──────────────┘               │              │
┌──────────────┐     HTTP      │              │
│  WebClient   │──────────────▶│              │
│  (порт 8083) │               └──────────────┘
└──────────────┘
```

**1. Сервис Source** — REST API с персистенцией на H2 (in-memory). Это наш «бэкенд», к которому обращаются остальные сервисы.

Следующие три сервиса решают одну и ту же задачу — управляют данными в Source — но используют разные технологии:

**2. Сервис RestClient** — использует `RestClient` (Spring Framework 6.1+). Это современная замена `RestTemplate` с fluent API. Синхронный, блокирующий — идеальный выбор для классических Spring MVC приложений.

**3. Сервис HTTP Interface** — использует декларативные HTTP-интерфейсы (Spring Framework 6.1+). Аналог OpenFeign, но встроенный в Spring — без внешних зависимостей. Вы описываете интерфейс, Spring генерирует реализацию.

**4. Сервис WebClient** — использует `WebClient` из Spring WebFlux. Неблокирующий, реактивный клиент. В этом уроке покажем его в реактивном стиле (возвращаем `Mono`/`Flux`), чтобы продемонстрировать разницу подходов.

Я специально сделал практически одинаковое описание для этих сервисов, потому что, как вы увидите ниже, они похожи как близнецы.

### TrackerService

```java
package ru.job4j.source.service;

import lombok.AllArgsConstructor;
import org.springframework.stereotype.Service;
import ru.job4j.source.model.Item;
import ru.job4j.source.repository.ItemRepository;
import java.util.List;

@Service
@AllArgsConstructor
public class TrackerService {
    private final ItemRepository repository;

    public Item save(Item item) {
        return repository.save(item);
    }

    public boolean update(int id, Item item) {
        boolean result = repository.existsById(id);
        if (result) {
            item.setId(id);
            repository.save(item);
        }
        return result;
    }

    public void delete(int id) {
        if (repository.existsById(id)) {
            repository.deleteById(id);
        }
    }

    public Item findById(int id) {
        return repository.findById(id).orElseThrow(IllegalArgumentException::new);
    }

    public List<Item> findByName(String name) {
        return repository.findByNameIgnoreCase(name);
    }

    public List<Item> findAll() {
        return (List<Item>) repository.findAll();
    }
}
```

### TrackerService

```java
package ru.job4j.restclient.service;

import lombok.AllArgsConstructor;
import org.springframework.core.ParameterizedTypeReference;
import org.springframework.http.HttpStatusCode;
import org.springframework.stereotype.Service;
import org.springframework.web.client.RestClient;
import ru.job4j.restclient.model.Item;
import java.util.Collections;
import java.util.List;

@Service
@AllArgsConstructor
public class TrackerService {

    private final RestClient client;

    public Item save(Item item) {
        return client.post()
                .body(item)
                .retrieve()
                .body(Item.class);
    }

    public boolean update(int id, Item item) {
        var response = client.put()
                .uri("?id={id}", id)
                .body(item)
                .retrieve()
                .toBodilessEntity();
        return response.getStatusCode().is2xxSuccessful();
    }

    public void delete(int id) {
        client.delete()
                .uri("?id={id}", id)
                .retrieve()
                .toBodilessEntity();
    }

    public Item findById(int id) {
        return client.get()
                .uri("/getById?id={id}", id)
                .retrieve()
                .body(Item.class);
    }

    public List<Item> findByName(String name) {
        return getList("/getByName?name={name}", name);
    }

    public List<Item> findAll() {
        return getList("/getAll");
    }

    private List<Item> getList(String uri, Object... uriVars) {
        List<Item> body = client.get()
                .uri(uri, uriVars)
                .retrieve()
                .body(new ParameterizedTypeReference<>() {});
        return body == null ? Collections.emptyList() : body;
    }
}
```

> **Ключевые отличия от RestTemplate:**
> - Fluent API: цепочка вызовов `.get().uri().retrieve().body()` вместо разрозненных методов `getForEntity`, `postForEntity`, `exchange`.
> - URI-шаблоны: `"?id={id}", id` вместо `String.format` / `.formatted()`.
> - Нет необходимости в `HttpEntity`, `HttpMethod` и других «обёрточных» классах.
> - `baseUrl` задан в конфигурации — в сервисе указываем только относительные пути.

Идея та же: вы описываете интерфейс с аннотациями, а Spring создаёт реализацию. Но теперь это часть core-фреймворка, без зависимости от Spring Cloud.

### TrackerProxy (HTTP Interface)

```java
package ru.job4j.httpinterface.service;

import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.service.annotation.DeleteExchange;
import org.springframework.web.service.annotation.GetExchange;
import org.springframework.web.service.annotation.PostExchange;
import org.springframework.web.service.annotation.PutExchange;
import ru.job4j.httpinterface.model.Item;
import java.util.List;

public interface TrackerProxy {

    @PostExchange
    Item save(@RequestBody Item item);

    @PutExchange
    ResponseEntity<Void> update(@RequestParam int id, @RequestBody Item item);

    @DeleteExchange
    void delete(@RequestParam int id);

    @GetExchange("/getById")
    Item getById(@RequestParam int id);

    @GetExchange("/getByName")
    List<Item> getByName(@RequestParam String name);

    @GetExchange("/getAll")
    List<Item> getAll();
}
```

> **Сравнение с OpenFeign:**
> - Аннотации `@GetExchange`, `@PostExchange` и т.д. вместо `@GetMapping`, `@PostMapping`.
> - Нет аннотации `@FeignClient` — привязка к URL происходит через конфигурацию.
> - Нет `@EnableFeignClients` — прокси создаётся вручную через `HttpServiceProxyFactory`.
> - Интерфейс выглядит почти идентично контроллеру Source — в этом и суть декларативного подхода.

### AppConfig

```java
package ru.job4j.httpinterface.config;

import org.springframework.beans.factory.annotation.Value;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.client.RestClient;
import org.springframework.web.client.support.RestClientAdapter;
import org.springframework.web.service.invoker.HttpServiceProxyFactory;
import ru.job4j.httpinterface.service.TrackerProxy;

@Configuration
public class AppConfig {

    @Value("${source-api-url}")
    private String url;

    @Bean
    public TrackerProxy trackerProxy() {
        RestClient restClient = RestClient.builder()
                .baseUrl(url)
                .build();
        RestClientAdapter adapter = RestClientAdapter.create(restClient);
        HttpServiceProxyFactory factory = HttpServiceProxyFactory
                .builderFor(adapter)
                .build();
        return factory.createClient(TrackerProxy.class);
    }
}
```

> Здесь мы создаём прокси на базе `RestClient` (синхронный). Можно точно так же использовать `WebClient` через `WebClientAdapter` — тогда вызовы будут реактивными.

### Почему в update нет проверки статус-кода?

Если вы сравните метод `update` в контроллерах RestClient и HTTP Interface, вы заметите разницу:

**RestClient — контроллер сам решает, какой статус вернуть:**

```java
// RestClient: сервис возвращает boolean
public boolean update(int id, Item item) {
    var response = client.put()...retrieve().toBodilessEntity();
    return response.getStatusCode().is2xxSuccessful();
}

// Контроллер: на основе boolean формирует ResponseEntity
boolean status = trackerService.update(id, item);
return ResponseEntity.status(status ? HttpStatus.OK : HttpStatus.NOT_FOUND).build();
```

**HTTP Interface — контроллер просто пробрасывает ответ Source как есть:**

```java
// Прокси: зеркалирует сигнатуру Source, возвращает ResponseEntity
@PutExchange
ResponseEntity<Void> update(@RequestParam int id, @RequestBody Item item);

// Контроллер: пробрасывает ResponseEntity напрямую
return trackerService.update(id, item);
```

Это не упущение — это следствие архитектуры. HTTP Interface (как и OpenFeign) работает по принципу «зеркала»: прокси-интерфейс повторяет контракт удалённого сервиса. Когда Source возвращает `200 OK` или `404 NOT_FOUND`, прокси оборачивает это в `ResponseEntity` и передаёт дальше. Контроллеру не нужно интерпретировать результат — он уже содержит правильный HTTP-статус от Source.

В RestClient же вы работаете на более низком уровне: сами формируете запрос, сами разбираете ответ, сами решаете, что вернуть клиенту. Это даёт больше контроля, но требует больше кода.

Можно сказать так: **RestClient** — это «я сам готовлю ужин», а **HTTP Interface** — «я заказываю доставку и передаю блюдо гостю как есть».

## Немного о реактивности

Прежде чем перейти к WebClient, давайте разберёмся, что такое **реактивное программирование**. Слово «реактивный» может вызвать ассоциации с ракетами и скоростью, но здесь оно означает другое — **реагирующий на события**. Как официант, который не стоит у кухни, ожидая ваше блюдо, а обслуживает другие столики и возвращается, когда блюдо готово.

### Синхронный (блокирующий) подход

Так работают RestClient и HTTP Interface — один поток обрабатывает один запрос от начала до конца:

```
Клиент          Сервис RestClient        Сервис Source
  │                    │                       │
  │── POST /item ─────▶│                       │
  │                    │── POST /api/item ────▶│
  │                    │                       │
  │                    │   поток ЗАБЛОКИРОВАН   │
  │                    │   ждёт ответа...       │
  │                    │   ничего не делает     │
  │                    │                       │
  │                    │◀── 200 OK ────────────│
  │◀── 200 OK ────────│                       │
```

**Модель: один запрос = один поток.** Пока Source обрабатывает запрос, поток в RestClient просто ждёт. Если придёт 200 одновременных запросов — нужно 200 потоков. Потоки — дорогой ресурс: каждый потребляет ~1 МБ памяти и время на переключение контекста. При 10 000 одновременных запросов такая модель задыхается.

### Реактивный (неблокирующий) подход

Так работает WebClient в связке с WebFlux:

```
Клиент          Сервис WebClient         Сервис Source
  │                    │                       │
  │── POST /item ─────▶│                       │
  │                    │── POST /api/item ────▶│
  │                    │                       │
  │                    │   поток СВОБОДЕН!      │
  │                    │   обрабатывает другие  │
  │                    │   запросы...           │
  │                    │                       │
  │                    │◀── 200 OK (событие) ──│
  │                    │   поток подхватывает   │
  │                    │   результат            │
  │◀── 200 OK ────────│                       │
```

**Модель: события, а не потоки.** Вместо блокировки потока WebClient регистрирует «подписку»: когда данные придут — вызови эту функцию. Освободившийся поток тем временем обрабатывает другие запросы. На 10 000 одновременных запросов может хватить 4–8 потоков.

### Когда это важно?

Реактивность **не делает один запрос быстрее**. Если Source отвечает за 50мс, и синхронный, и реактивный клиент получат ответ через 50мс. Выигрыш появляется при **высокой конкурентности** — когда сервис обслуживает тысячи одновременных запросов. Реактивный подход потребляет меньше памяти и не страдает от исчерпания пула потоков.

Но у него есть цена: код сложнее читать и отлаживать. Вместо привычного `Item item = service.findById(1)` вы работаете с `Mono<Item>` — контейнером, который *когда-нибудь* будет содержать результат. Стектрейсы становятся менее информативными, а отладка — менее интуитивной.

**Правило выбора простое:**
- Ваш сервис обрабатывает сотни запросов в секунду и не упирается в ресурсы → RestClient / HTTP Interface
- Тысячи одновременных соединений, стриминг данных, event-driven архитектура → WebClient + WebFlux

### TrackerService

```java
package ru.job4j.webclient.service;

import lombok.AllArgsConstructor;
import org.springframework.stereotype.Service;
import org.springframework.web.reactive.function.client.WebClient;
import reactor.core.publisher.Flux;
import reactor.core.publisher.Mono;
import ru.job4j.webclient.model.Item;

@Service
@AllArgsConstructor
public class TrackerService {

    private final WebClient client;

    public Mono<Item> save(Item item) {
        return client.post()
                .bodyValue(item)
                .retrieve()
                .bodyToMono(Item.class);
    }

    public Mono<Boolean> update(int id, Item item) {
        return client.put()
                .uri("?id={id}", id)
                .bodyValue(item)
                .retrieve()
                .toBodilessEntity()
                .map(response -> response.getStatusCode().is2xxSuccessful());
    }

    public Mono<Void> delete(int id) {
        return client.delete()
                .uri("?id={id}", id)
                .retrieve()
                .toBodilessEntity()
                .then();
    }

    public Mono<Item> findById(int id) {
        return client.get()
                .uri("/getById?id={id}", id)
                .retrieve()
                .bodyToMono(Item.class);
    }

    public Flux<Item> findByName(String name) {
        return client.get()
                .uri("/getByName?name={name}", name)
                .retrieve()
                .bodyToFlux(Item.class);
    }

    public Flux<Item> findAll() {
        return client.get()
                .uri("/getAll")
                .retrieve()
                .bodyToFlux(Item.class);
    }
}
```

> **Ключевое отличие от старого урока:** методы возвращают `Mono<T>` и `Flux<T>` вместо обычных объектов. Нет ни одного вызова `.block()`. Это правильное использование WebClient — полностью неблокирующее. Подписка на результат происходит на уровне WebFlux-контроллера.
>
> В старом уроке WebClient использовался с `.block()`, что фактически превращало его в синхронный клиент и убивало весь смысл реактивности. Если вам нужен синхронный вызов — используйте `RestClient`, а не `WebClient` с `.block()`.

### TrackerController

```java
package ru.job4j.webclient.controller;

import lombok.AllArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;
import reactor.core.publisher.Flux;
import reactor.core.publisher.Mono;
import ru.job4j.webclient.model.Item;
import ru.job4j.webclient.service.TrackerService;

@AllArgsConstructor
@RestController
@RequestMapping("/webclient")
public class TrackerController {
    private final TrackerService trackerService;

    @PostMapping
    public Mono<Item> save(@RequestBody Item item) {
        return trackerService.save(item);
    }

    @PutMapping
    public Mono<ResponseEntity<Void>> update(@RequestParam int id,
                                              @RequestBody Item item) {
        return trackerService.update(id, item)
                .map(success -> ResponseEntity
                        .status(success ? HttpStatus.OK : HttpStatus.NOT_FOUND)
                        .<Void>build());
    }

    @DeleteMapping
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public Mono<Void> delete(@RequestParam int id) {
        return trackerService.delete(id);
    }

    @GetMapping("/getById")
    public Mono<Item> getById(@RequestParam int id) {
        return trackerService.findById(id);
    }

    @GetMapping("/getByName")
    public Flux<Item> getByName(@RequestParam String name) {
        return trackerService.findByName(name);
    }

    @GetMapping("/getAll")
    public Flux<Item> getAll() {
        return trackerService.findAll();
    }
}
```

> Контроллер тоже реактивный — возвращает `Mono` и `Flux`. Spring WebFlux подписывается на них автоматически и отправляет ответ клиенту, когда данные будут готовы.

## Сравнительная таблица

| Критерий | RestClient | HTTP Interface | WebClient |
|----------|-----------|----------------|-----------|
| **Стиль** | Императивный, fluent API | Декларативный (интерфейс) | Реактивный, fluent API |
| **Блокирование** | Синхронный | Зависит от адаптера* | Неблокирующий |
| **Внешние зависимости** | Нет (spring-web) | Нет (spring-web) | spring-webflux |
| **Появился в** | Spring 6.1 / Boot 3.2 | Spring 6.1 / Boot 3.2 | Spring 5.0 / Boot 2.0 |
| **Когда использовать** | Spring MVC, простые вызовы | Много эндпоинтов, зеркалирование API | Реактивный стек, стриминг |
| **Объём кода** | Средний | Минимальный | Средний |

\* HTTP Interface может работать как через `RestClientAdapter` (синхронно), так и через `WebClientAdapter` (реактивно).

## Что выбрать?

**Новый проект на Spring MVC** → `RestClient`. Это прямая, рекомендованная Spring замена `RestTemplate`.

**Много эндпоинтов у целевого сервиса, хотите минимум кода** → HTTP Interface. Описываете интерфейс — Spring делает остальное.

**Реактивный стек (WebFlux), стриминг, высокая конкурентность** → `WebClient`. Используйте его по-настоящему реактивно, без `.block()`.

**Легаси с RestTemplate** → миграция на `RestClient` максимально простая, API похож по духу, но чище. Планируйте миграцию до 2028 года.

**OpenFeign** → всё ещё работает, но для новых проектов предпочтительнее HTTP Interface, так как он не требует Spring Cloud.
