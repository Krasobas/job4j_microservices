# 1. Сага на основе хореографии [v2, 2026]

## У каждого сервиса своя БД

Прежде чем разбирать саги, нужно понять ключевую особенность микросервисной архитектуры: **у каждого сервиса своя база данных**. Это не рекомендация, а архитектурный принцип — паттерн [Database per Service](https://microservices.io/patterns/data/database-per-service.html).

> **Терминология.** В оригинальном уроке использовался термин «повествование» (дословный перевод Saga). В индустрии закрепился термин **сага** — его и будем использовать. Оба слова означают одно и то же.

Почему у каждого сервиса своя БД:

1. **Производительность.** Запросы распределяются между разными инстансами БД, а не конкурируют за одну.
2. **Независимость разработки.** Команды работают со своими базами, не мешая друг другу миграциями.
3. **Свобода выбора технологии.** Один сервис может использовать PostgreSQL, другой — MongoDB, третий — Redis. Каждому — подходящий инструмент.
4. **Фиксация состояния.** Каждый сервис контролирует состояние своей части системы. Например, сервис заказов отслеживает жизненный цикл заказа: СОЗДАН → ПОДТВЕРЖДЁН → ОТПРАВЛЕН → ДОСТАВЛЕН.

## Проблема: распределённые транзакции

В монолите транзакция обеспечивает ACID (Atomicity, Consistency, Isolation, Durability). Одна БД, один `@Transactional` — и все изменения либо применились, либо откатились.

Но когда бизнес-операция затрагивает несколько сервисов с **разными** БД, классическая транзакция не работает:

```
Заказ еды:
  1. OrderService  → создать заказ        (PostgreSQL)
  2. PaymentService → списать деньги       (PostgreSQL)
  3. KitchenService → начать готовить       (MongoDB)
  4. DeliveryService → назначить курьера    (PostgreSQL)

Один @Transactional на 4 базы? Нет.
```

Для решения этой проблемы используются **саги**.

## Что такое сага

**Сага** — это паттерн, при котором распределённая бизнес-операция реализуется как **цепочка локальных транзакций**, связанных событиями.

- **Локальная транзакция** — обычная ACID-транзакция внутри одного сервиса, в одной БД.
- **Событие** — сообщение от одного сервиса другим: «я сделал своё дело, твой ход». Передаётся через брокер сообщений (Kafka, RabbitMQ).
- **Компенсация** — если на каком-то шаге произошёл сбой, предыдущие шаги откатываются через компенсирующие транзакции (не `ROLLBACK`, а отдельные действия: «отменить заказ», «вернуть деньги»).

## Что такое хореография

Саги реализуются двумя способами. В этом уроке — **хореография**.

**Хореография** — децентрализованный подход, при котором сервисы **сами** решают, как реагировать на события других сервисов. Нет центрального координатора — каждый сервис слушает события и действует самостоятельно.

Аналогия: танцоры на сцене знают свои партии и реагируют на движения друг друга, без дирижёра.

## Пример: заказ еды

```
OrderService          PaymentService         KitchenService
     │                      │                      │
     │── ORDER_CREATED ────▶│                      │
     │                      │                      │
     │                      │── PAYMENT_SUCCESS ──▶│
     │                      │                      │
     │                      │                      │── COOKING_STARTED
     │                      │                      │
     │◀── COOKING_DONE ─────┼──────────────────────│
     │                      │                      │
     │── ORDER_COMPLETED    │                      │
```

Каждый сервис:
1. Получает событие
2. Выполняет **локальную транзакцию** в своей БД
3. Публикует **новое событие** для следующего участника

Если PaymentService не смог списать деньги:

```
OrderService          PaymentService
     │                      │
     │── ORDER_CREATED ────▶│
     │                      │
     │◀── PAYMENT_FAILED ───│
     │                      │
     │── ORDER_CANCELLED    │  ← компенсирующая транзакция
```

## Плюсы и минусы хореографии

**Плюсы:**
- Простота для небольшого числа шагов (2–3 сервиса)
- Нет единой точки отказа (нет координатора)
- Слабая связанность — сервисы знают только о событиях, не друг о друге

**Минусы:**
- Сложно отследить, на каком этапе находится сага (нет единого места)
- При 5+ сервисах цепочки событий становятся запутанными
- Циклические зависимости — сервис A слушает B, B слушает C, C слушает A

Для сложных саг лучше подходит оркестрация — следующий урок.
