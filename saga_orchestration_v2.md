# 2. Сага на основе оркестрации [v2, 2026]

## Что такое оркестрация

В предыдущем уроке мы рассмотрели хореографию — децентрализованный подход, где сервисы сами реагируют на события друг друга. Оркестрация — противоположный подход: один **оркестратор** управляет всей сагой.

**Оркестратор** — это компонент внутри одного из сервисов (не отдельный сервис!), который отправляет **команды** другим сервисам и принимает от них **ответы**. На основании ответов — отправляет следующие команды или запускает компенсацию.

Аналогия: дирижёр оркестра. Музыканты не решают сами, когда вступать — дирижёр даёт команды.

```
                    ┌──────────────┐
          команда   │              │   команда
       ┌────────────│  Оркестратор │────────────┐
       │            │  (в CreditSvc)│            │
       │            └──────┬───────┘            │
       ▼                   │                    ▼
┌─────────────┐      ответ │           ┌──────────────┐
│  Document   │◀───────────┘──────────▶│ CreditHistory│
│  Service    │────────────▶           │  Service     │
└─────────────┘    ответ               └──────────────┘

─────▶  команда (оркестратор говорит ЧТО делать)
- - -▶  ответ (сервис говорит ЧТО сделано)
```

## Пример: одобрение кредита

**Сервисы:**
1. **CreditService** — принимает запрос от клиента. Содержит оркестратор.
2. **DocumentService** — проверяет документы.
3. **CreditHistoryService** — проверяет кредитную историю.

**Последовательность:**

```
Клиент        CreditService         DocumentSvc      CreditHistorySvc
  │           (оркестратор)              │                   │
  │── запрос ──▶│                        │                   │
  │             │                        │                   │
  │             │── ПРОВЕРЬ_ДОКУМЕНТЫ ──▶│                   │
  │             │                        │                   │
  │             │◀── ДОКУМЕНТЫ_ОК ───────│                   │
  │             │                        │                   │
  │             │── ПРОВЕРЬ_ИСТОРИЮ ─────┼──────────────────▶│
  │             │                        │                   │
  │             │◀── ИСТОРИЯ_ОК ─────────┼───────────────────│
  │             │                        │                   │
  │◀── ОДОБРЕН ─│                        │                   │
```

Если документы не прошли проверку — оркестратор **сразу** возвращает отказ, не обращаясь к CreditHistoryService. Это преимущество: оркестратор знает полную картину и принимает решения в одном месте.

## Компенсация при оркестрации

Если на шаге 3 CreditHistoryService нашёл проблемы, оркестратор запускает компенсацию:

```
Оркестратор          DocumentSvc         CreditHistorySvc
     │                    │                      │
     │── ПРОВЕРЬ_ДОКУМЕНТЫ ─▶│                   │
     │◀── ДОКУМЕНТЫ_ОК ──────│                   │
     │                                           │
     │── ПРОВЕРЬ_ИСТОРИЮ ───────────────────────▶│
     │◀── ИСТОРИЯ_ПЛОХАЯ ───────────────────────│
     │                                           │
     │── ОТМЕНИ_ПРОВЕРКУ ────▶│                  │  ← компенсация
     │◀── ОТМЕНЕНО ───────────│                  │
     │                                           │
     │── КРЕДИТ_ОТКЛОНЁН                         │
```

Оркестратор **точно знает**, какие шаги были выполнены, и откатывает их в обратном порядке. В хореографии это было бы сложнее — каждый сервис должен сам решать, когда откатываться.

## Плюсы и минусы оркестрации

**Плюсы:**
- **Централизованная логика.** Вся бизнес-логика саги в одном месте — легко понять, отладить, протестировать.
- **Упрощённые зависимости.** Сервисы зависят только от оркестратора, а не друг от друга.
- **Прозрачная компенсация.** Оркестратор знает, какие шаги выполнены, и может откатить их в правильном порядке.
- **Масштабируется на сложные саги.** 5, 10, 15 шагов — оркестратор справляется, хореография — нет.

**Минусы:**
- **Единая точка ответственности.** Оркестратор становится критически важным компонентом. Если он упал — вся сага остановилась.
- **Риск «умного» оркестратора.** Если оркестратор начинает выполнять бизнес-логику вместо того, чтобы просто координировать — это антипаттерн. Оркестратор должен **командовать**, а не **делать**.

## Когда что использовать

| Критерий | Хореография | Оркестрация |
|----------|-------------|------------|
| **Число участников** | 2–3 сервиса | 4+ сервисов |
| **Сложность логики** | Линейная цепочка | Ветвления, условия |
| **Компенсация** | Каждый сервис сам | Оркестратор управляет |
| **Связанность** | Слабая (события) | Средняя (команды) |
| **Отладка** | Сложно | Просто (одно место) |
| **Единая точка отказа** | Нет | Оркестратор |
